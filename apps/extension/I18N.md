# Extension UI internationalization (i18n)

The extension **chrome** (popup, onboarding, pill, settings panel, consent
prompt, definition card) renders in the user's browser UI language. The subtitle
**output** was already language-universal — this is purely interface text.

Shipped 2026-07-11. Locales: **en** (canonical) + **ja zh ko de fr es it uk ru**.
Any other browser language falls back to English.

## Architecture (framework-free, module singleton)

`lib/i18n/`:
- **`resolve.ts`** — `resolveUiLocale(code?)` maps a BCP-47 code (or
  `browser.i18n.getUILanguage()`, guarded) to one of the 10 UI locales. All
  Chinese variants → `zh` (Simplified; a `zh-Hant` table can be added later).
- **`strings/en.ts`** — the **single source of truth**: 143 keys with English
  values and `{param}` placeholders. Every locale table is a `Partial` of it, so
  a missing key falls back to English (a partial translation is safe to ship).
- **`strings/<loc>.ts`** — one `LocaleTable` per locale.
- **`index.ts`** — `initUiLocale()`, `t(key, params?)`, `languageName(code)`,
  `getUiLocale()`.

The UI locale is **fixed at load** (it's the browser language), so `t()` is a
module singleton — no React context, no provider, no prop-drilling. Components
just `import { t } from "@/lib/i18n"`.

`initUiLocale()` is called once per entrypoint:
- popup / onboarding — in their `main.tsx`, before render.
- in-page overlay (all 5 content scripts) — at module load in
  `components/loom-app.tsx` (every content script renders `LoomApp`).

## Rules when adding or changing UI strings

1. **Add the key to `en.ts` first.** Then translate it in the 9 locale files (or
   leave it — English shows through until translated). `i18n.test.ts` asserts
   every locale has exactly the canonical key set and that every `{placeholder}`
   survives translation, so drift fails CI.
2. **Never call `t()` at module top level.** `initUiLocale()` runs *after*
   module-level consts are evaluated, so a `t()` in a top-level const resolves to
   English. Const label-tables that need translation are built by a **render-time
   function** instead (see `supportedLangs()`, `positionOptions()`,
   `fontFamilyOptions()`, `longVowelOptions()` in `settings-panel.tsx`). `t()`
   inside a component/callback/function body is fine.
3. **Language display names come from `languageName(code)`** (Intl.DisplayNames),
   never a hand-translated table — the 49-entry picker localizes for free.
4. **Proper nouns stay literal in the components, not in `en.ts`:** platform
   names (YouTube/Netflix/…), romanization systems (Pinyin/Zhuyin/Jyutping/
   Romaji/IPA/RTGS/DIN 31635/ALA-LC/DMG/Hunterian), Noto font names, the "Loom"
   wordmark, and the `asr` track badge.

## Not yet localized (deliberate)

- The **manifest** (`name` / `description` / `default_title` in `wxt.config.ts`)
  — would need `_locales/` + `__MSG_*`; the in-page + popup chrome is the
  user-visible surface. Low priority.
- **Preset / group labels + descriptions** come from the presets API
  (`lib/presets/`), and the alt-orthography `VariantDescriptor` hints come from
  `@loom/orthography-tables` — both are server/package data, localized at source
  if ever.
- A **Traditional Chinese** UI table (`zh-Hant`) — all Chinese currently → `zh`.
